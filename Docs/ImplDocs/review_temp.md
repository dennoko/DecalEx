# DecalEx パフォーマンスレビュー（日本語暫定版）

日付: 2026-03-15
対象: `Shaders/custom.hlsl`（実行時ピクセルシェーダー経路）、静的コードレビューのみ（GPUプロファイラ未取得）

## 要約

現在の実装は機能が充実して構造も良好ですが、複数機能を同時に有効にするとフラグメント経路のコストが高くなり得ます。

最も影響の大きい改善候補は次のとおりです。

1. 回転UVやZロールキャンセルのための三角関数・基底再計算をピクセル毎に行わない。
2. デカール範囲外での不要なテクスチャフェッチを回避する。
3. 実行時分岐でブレンドモードを選ぶのではなく、可能ならキーワード/コンパイル時分岐に置き換える。

最悪ケース（全モジュール有効）では、`BEFORE_MATCAP` 内だけで概ねピクセルあたり14サンプル程度、加えて多くのALU重めの分岐や `pow()` 呼び出しが発生します。

---

## 発見事項（深刻度順）

### [高] 複数モジュールで三角関数計算がピクセル毎に重複

- 根拠:
  - `DNKW_DecalMaskUV` が呼び出し毎に `sincos()` を実行: `Shaders/custom.hlsl:169`, `Shaders/custom.hlsl:174`
  - 呼び出し箇所が複数存在: `Shaders/custom.hlsl:222`, `Shaders/custom.hlsl:280`, `Shaders/custom.hlsl:291`, `Shaders/custom.hlsl:386`
- 影響:
  - `sincos` は高コストな演算であり、ピクセル毎・モジュール毎に実行されるとALU負荷が著しく増加します。
  - 複数機能が同時に有効な場合、同じ角度変換がフラグメント内で何度も繰り返されます。
- 改善案:
  - 角度の `sin/cos` をCPU側で事前計算し、定数バッファ（CBUFFER）やパックされた回転パラメータで渡す。
  - ランタイムの三角関数を、事前計算した値を用いる2x2回転行列乗算に置き換える。
  - 見た目に許容できるなら角度更新を量子化してパラメータの変化頻度を下げる。

### [高] Zロールキャンセルの基底を MatCap ごとにピクセル毎に再構築している

- 根拠:
  - ベース MatCap の Z-roll パス: `Shaders/custom.hlsl:197`-`Shaders/custom.hlsl:207`
  - Slot1 MatCap: `Shaders/custom.hlsl:329`-`Shaders/custom.hlsl:335`
  - Slot2 MatCap: `Shaders/custom.hlsl:424`-`Shaders/custom.hlsl:430`
  - `cross`・`length`・`normalize` 系のカメラ由来ベクトル処理が繰り返されている。
- 影響:
  - カメラ基底はフレーム単位のデータであるため、ピクセル毎に再計算するのは不必要なALUコストを生む。
- 改善案:
  - ロールキャンセル用の基底をフレーム単位（グローバルなシェーダーパラメータ）または頂点単位で事前計算して渡す。
  - 複数の MatCap ブロック間で基底を共有して再利用する。

### [高] デカール範囲外のピクセルでも Slot1/Slot2 の高コストなサンプリングが実行される

- 根拠:
  - 範囲判定は行われているが: `Shaders/custom.hlsl:292`, `Shaders/custom.hlsl:387`
  - その後のテクスチャフェッチが無条件に行われている:
    - Slot1 の normal/decal フェッチ: `Shaders/custom.hlsl:299`, `Shaders/custom.hlsl:304`
    - Slot2 の normal/decal フェッチ: `Shaders/custom.hlsl:394`, `Shaders/custom.hlsl:399`
- 影響:
  - 小さいデカールが大きなメッシュに投影される状況では、デカール非適用ピクセルにも余計なサンプルコストがかかる。
- 改善案:
  - 高コスト処理の前に早期判定（例えば shared mask が小さければ処理をスキップ）を追加する。
  - ブランチのコヒーレンスに注意しつつ、デカール被覆がまばらなケースで効果が出るようにする。

### [中] ブレンドモードのランタイムチェーンがフラグメント経路で長い分岐を生む

- 根拠:
  - MatCap のブレンドチェーン: `Shaders/custom.hlsl:251`-`Shaders/custom.hlsl:266`
  - Slot1 のデカールブレンド開始: `Shaders/custom.hlsl:308`-`Shaders/custom.hlsl:309`
  - Slot2 のデカールブレンド開始: `Shaders/custom.hlsl:403`-`Shaders/custom.hlsl:404`
- 影響:
  - ブレンドモードはマテリアル定数であることが多いため、ランタイム `if/else` をピクセル毎に実行するのは命令数増大につながる。
- 改善案:
  - 頻用されるブレンドモードについては `shader_feature_local` / マルチコンパイルでの分岐（コンパイル時）を検討する。
  - バリアント数を増やせない場合は、よく使うブレンドに対して分岐深度を下げる最適化を行う。

### [中] モジュール有効/無効がランタイム中心のため無効時も分岐コストが残る

- 根拠:
  - `BEFORE_MATCAP` に全モジュールのマクロ呼び出しが含まれる: `Shaders/custom.hlsl:481`-`Shaders/custom.hlsl:486`
- 影響:
  - ほとんどの機能を無効にしている場合でも、各ピクセルで有効判定の分岐コストが発生する。
- 改善案:
  - MatCap/Emission/Slot1/Slot2 のようなトップレベル機能はキーワード駆動（コンパイル時）にして、未使用経路をバイナリから除外できるようにする。

### [低] 全機能有効時のサンプル数が高くなる可能性

- 根拠:
  - MatCap: 最大で2テクスチャ（MatCap + mask）: `Shaders/custom.hlsl:217`, `Shaders/custom.hlsl:224`
  - Emission: 最大2（emission + mask）: `Shaders/custom.hlsl:276`, `Shaders/custom.hlsl:282`
  - 各スロットは最大5（mask + normal + decal + matcap + emission）程度をサンプリングする可能性がある（該当箇所を参照）
- 影響:
  - ベースの lilToon コストに加え、最悪でおよそピクセルあたり14サンプルに到達する。
- 改善案:
  - モバイル/デスクトップなど品質帯を定め、マテリアルプロファイルで同時有効機能の上限を設定する。

---

## 推奨する最適化実行順

1. 角度の三角関数とZ-roll基底のピクセル毎計算を除去する。
2. 範囲外ピクセルの早期スキップを導入して高コストフェッチを回避する。
3. 主要モジュールと汎用ブレンドをコンパイル時分岐で特殊化する。
4. Unity Frame Debugger / RenderDoc / PIX などで代表ケースをプロファイルして効果を検証する。

---

## 注意事項

- 本レビューは静的解析に基づくため、最終的な優先度はターゲット機器（Quest / PCVR 等）でのGPUプロファイリング結果で確定してください。
- レビューはコードを変更していないため、挙動上の回帰は発生していません。
